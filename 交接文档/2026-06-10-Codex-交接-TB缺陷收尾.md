## 失败的尝试

- ❌ 直接用旧入口文本创建缺陷 → 因为 `scripts/call_ci_assistant.py` 会把“结果”类词误判成查询意图，走成 `bug.query`，返回“没有找到最近的 bug 创建记录”。
- ❌ 用户在钉钉里确认创建后第一次真实提交失败 → 因为 `_sprintId` 被传成了中文 `线上缺陷迭代`，Teambition 返回参数错误，提示它应当是 `null` 类型。
- ❌ 早期真实创建虽然能成功，但页面里带了很多不该自动填的字段 → 因为 payload 直接复用了 HAR 模板里残留的整包 `customfields`。
- ❌ 本地 shell 里直接写中文字面量做回归时，部分测试任务标题和备注变成 `?` → 这是 PowerShell / shell 注入中文方式导致的本地测试问题，不是 Teambition Web API 本身问题。
- ❌ 本地第一次用中文字面量测试 sprint 匹配时，误以为“线上缺陷迭代”匹配不上 → 实际是 shell 里的中文在脚本入参阶段就被变成了 `?`，是假阴性。

## 关键决策

- 决策：继续保留主流程里的确认门，原因：这是当前唯一能避免误创建真实公司缺陷的安全门。
- 决策：`sprint` 字段必须在本地映射成真实 id 后再发，原因：Teambition 不接受中文迭代名直接写进 `_sprintId`。
- 决策：不要再沿用抓包模板里的整包 `customfields`，原因：模板里有大量历史默认值和残留选项，会污染真实缺陷表单。
- 决策：当前只清理本轮明确看到的模板残留问题，不继续扩展更多猜测字段规则，原因：用户要求只做必要修改，不要堆补丁和规则。
